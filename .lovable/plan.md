
## Round 9 Bug Hunt — Complete Analysis

After thoroughly reading all 8 target files (CheckoutContraoferta, CheckoutCotizacion, ComoComprar, ComoVender, Cotizador, SubastasYOfertas, mercadopago-webhook, envia-quote) plus Subastas, MisOfertas, useOffers, ShippingQuote, and App.tsx, here are all bugs found:

### Bug 1 (HIGH): `CheckoutContraoferta.tsx` — Order created without `order_number` guard — if DB trigger doesn't exist yet, `createdOrder.order_number` will be `null` and SPEI reference shown to user is blank

Line 93 comment says `// Create order — order_number is generated by DB trigger set_order_number` but the DB trigger does NOT exist (confirmed from `<db-triggers>`: "There are no triggers in the database"). So `order.order_number` comes back `null`. 

On line 353: `Pedido #{createdOrder?.order_number}` displays as "Pedido #" (empty).
On line 198: `navigate('/checkout/success?order=${createdOrder.order_number}')` passes `?order=null`.
On line 399: SPEI reference field shows `createdOrder?.order_number || ''` — shows empty string.

The customer sees no order number in the SPEI transfer instructions, so they have no reference to put in their bank transfer. This is a critical UX/operational failure.

**Fix:** This is the same issue as CheckoutSuccess (Round 6). The root fix is the missing DB trigger. As a client-side patch: generate a fallback order number if null using the same `MI-YYYYMMDD-NNNN` format pattern used in `mercadopago-create-preference` after Round 7 fix. But the real fix is: create the DB trigger for `generate_order_number()` which already exists as a DB function.

**DB trigger needed (this was noted in prior rounds too):**
```sql
CREATE TRIGGER set_order_number
BEFORE INSERT ON public.orders
FOR EACH ROW
EXECUTE FUNCTION public.generate_order_number();
```

### Bug 2 (HIGH): `CheckoutContraoferta.tsx` — SPEI "Ya realicé la transferencia" button has NO validation that the SPEI reference number field is populated — user can click without entering anything

In `CheckoutContraoferta.tsx` SPEI section (lines 386–449), there is NO `<Input>` field for a SPEI reference number at all — unlike `CheckoutCotizacion.tsx` which has a `speiReference` state and requires it before the button is enabled (line 528: `disabled={isProcessing || !speiReference.trim()}`).

In `CheckoutContraoferta.tsx`, the SPEI section shows bank data but the "Ya realicé la transferencia" button (line 443) has NO reference number collection and no disable guard — it's immediately clickable. The admin has no way to match this SPEI payment to a bank statement.

**Fix:** Add a `speiReference` state and an `<Input>` field for the SPEI folio/reference in `CheckoutContraoferta.tsx`, same pattern as `CheckoutCotizacion.tsx`. Disable the confirm button until a reference is entered. Also store the reference in the order's `notes` field.

### Bug 3 (HIGH): `mercadopago-webhook/index.ts` — `MERCADOPAGO_WEBHOOK_SECRET` is not in the secrets list, so the signature validation always falls back to `return true` — webhook accepts any unauthenticated POST

From `<secrets>` section: the secrets are `MERCADOPAGO_ACCESS_TOKEN`, `LOVABLE_API_KEY`, `RESEND_API_KEY`, `ENVIA_API_KEY` — there is **NO `MERCADOPAGO_WEBHOOK_SECRET`** secret.

At line 18–22:
```ts
const WEBHOOK_SECRET = Deno.env.get('MERCADOPAGO_WEBHOOK_SECRET');
if (!WEBHOOK_SECRET) {
  console.warn('MERCADOPAGO_WEBHOOK_SECRET not set — skipping signature validation');
  return true; // ← accepts ANY POST with x-signature header present!
}
```

Actually even worse: line 13 checks `if (!xSignature) return false` — but if ANY `x-signature` header is present (which an attacker can trivially add), AND `MERCADOPAGO_WEBHOOK_SECRET` is not set, the function returns `true`. The backward-compat fallback means the webhook can be called by any attacker who knows the URL and adds an `x-signature` header. This lets them forge fake `approved` payment statuses and mark orders as `paid`.

**Fix:** The secret needs to be set. Since we can't set it here (it requires the actual MercadoPago webhook secret from the MP dashboard), note this as a critical operational security gap. As a code-level defense: change the fallback from `return true` to `return false` when the secret is missing, or at minimum log prominently. Then add a note to configure the secret.

### Bug 4 (MEDIUM): `envia-quote/index.ts` — logs the full API key length and prefix to console — information disclosure in production logs

Lines 38–39:
```ts
console.log('API Key length:', ENVIA_API_KEY.length);
console.log('API Key prefix:', ENVIA_API_KEY.substring(0, 10) + '...');
```
These are debug logs left in production code. Supabase edge function logs are visible to admins with dashboard access. While the key isn't fully exposed, leaking the first 10 characters + length narrows the key space significantly if logs are compromised.

**Fix:** Remove the API key debug logs. Keep the "Using Envia API URL:" log if needed.

### Bug 5 (MEDIUM): `envia-quote/index.ts` — fires 13 parallel requests (one per carrier) with no timeout — a slow/hanging carrier request can block the entire response for 30+ seconds

Lines 170–198: All 13 carrier requests are fired with `Promise.all`. If one carrier's API hangs (no response, not a timeout error), the entire `Promise.all` waits until Deno's fetch default timeout (which can be very long). The function could time out at the edge function level (Supabase default: 60s), returning a 504 to the user's shipping quote widget.

**Fix:** Add per-request timeouts using `AbortController`:
```ts
const controller = new AbortController();
const timeout = setTimeout(() => controller.abort(), 8000); // 8s per carrier
const response = await fetch(url, { signal: controller.signal, ... });
clearTimeout(timeout);
```
This ensures no single carrier can hold up the entire quote.

### Bug 6 (MEDIUM): `SubastasYOfertas.tsx` — `useOffers({ userId: user?.id })` fetches with no `.limit()` — loads ALL user offers including historical ones

Line 29: `const { data: userOffers } = useOffers({ userId: user?.id });`

Looking at `useOffers` in `useOffers.ts` (lines 8–29): the query has NO `.limit()`. A user with many historical offers (accepted, rejected, etc.) will load all of them into this info/landing page. This is low impact individually but follows the same pattern fixed in Round 6 for `useAdminOffers`.

**Fix:** Add `.limit(10)` to `useOffers` when `userId` is provided, or add a `limit` option to the hook.

### Bug 7 (MEDIUM): `Subastas.tsx` — `statusFilter === 'scheduled'` filter maps to `auction_status: 'inactive'` instead of `'scheduled'`

Lines 32–33:
```ts
} else if (statusFilter === 'scheduled') {
  query = query.eq('auction_status', 'inactive');
}
```
The dropdown label says "Próximas" (scheduled auctions) but filters for `auction_status = 'inactive'`. In `AuctionSection.tsx`, products start with `auction_status: 'inactive'` and transition through `'scheduled' → 'active' → 'ended_valid'/'ended_invalid'/'sold'`.

If "scheduled" is a valid `auction_status` value (products explicitly marked as scheduled to start at a future date), they should be filtered with `.eq('auction_status', 'scheduled')`, not `'inactive'`. Filtering for `'inactive'` shows all products not yet set up for auction, not future ones.

**Fix:** Change the filter to `.eq('auction_status', 'scheduled')` to correctly show only scheduled/upcoming auctions.

The admin never sees what SPEI folio the customer provided.

### Bug 10 (LOW): `envia-quote` — destination city and state fields in the payload are hardcoded to "Ciudad de México" / "DF" regardless of the actual `zipTo` provided

Lines 129–140:
```ts
destination: {
  city: "Ciudad de México",  // ← hardcoded
  state: "DF",               // ← hardcoded
  postalCode: zipTo,         // ← correct
}
```
The Envia API uses both the city/state AND the postalCode for routing. If the destination is Monterrey (CP 64000) but city is hardcoded to "Ciudad de México" / "DF", some carriers might reject the address or return incorrect rates. This could cause certain routes to show no results or wrong pricing.

**Fix:** While we can't know the destination city/state from a zip code alone (would need a reverse postal lookup), the minimal fix is to remove the misleading hardcoded values and use generic placeholders like `"N/A"` or use an empty string if the API allows it, OR use the Envia API to lookup city from postalCode first.
